# 웹 보안 취약점 개선 및 세션 인증 강화 구현 계획서 v1.0

보안 진단을 통해 파악된 핵심 취약점(서버 액션 권한 검증 부재 및 클라이언트 단 쿠키 변조 가능성)을 해결하기 위해, Next.js의 표준 `cookies()` API와 데이터베이스 토큰 검증 시스템을 연동한 **보안 세션 메커니즘**을 구현합니다.

## 핵심 변경점 (Best Practices)
1. **쿠키 보안 강화 (`httpOnly`, `secure`, `sameSite` 적용)**:
   - 브라우저 자바스크립트(`document.cookie`)로 가짜 이메일 값을 쿠키에 변조 삽입해 권한을 획득하던 기존 방식을 완전히 제거합니다.
   - 서버 세션 확인용 랜덤 토큰인 `session_token`만 암호화/보안 설정(`httpOnly`)이 완료된 쿠키로 전송하여 클라이언트 측 위변조 및 XSS 탈취 공격을 차단합니다.
2. **서버 사이드 권한 가드(Guard) 적용**:
   - 모든 중요 데이터 접근/수정 서버 액션 내부에서 호출 주체의 `session_token`이 유효한지 데이터베이스를 거쳐 무조건 검증합니다.
   - 담당자별로 본인이 담당하는 예약 목록만 보거나(BOLA 방어), 등록된 보안 협력사/최고 관리자만 특정 기능에 접근하도록 통제(BFLA 방어)합니다.

## Proposed Changes

### [Backend Component]

보안 검증 및 세션 관리를 담당할 공통 가드 모듈을 설계하고, 기존 서버 액션 함수들에 검증 로직을 적용합니다.

#### [NEW] [auth.ts](file:///e:/visit/src/lib/auth.ts)
- `cookies()` API를 통해 현재 쿠키에 저장된 `session_token`을 읽습니다.
- DB의 `SecurityToken` 테이블을 조회하여 해당 토큰의 소유자(이메일), 만료일 등을 검증합니다.
- 토큰이 없거나 만료된 경우 `Error("Unauthorized")`를 발생시켜 서버 액션 실행을 즉각 차단합니다.
- 특정 권한(예: 보안 협력사 회원 여부, 호스트 본인 여부)을 검증하는 헬퍼 함수들을 제공합니다.

#### [MODIFY] [reservation.ts](file:///e:/visit/src/app/actions/reservation.ts)
- `verifyHostToken` / `verifySecurityToken` 실행 성공 시, Next.js `cookies` API를 호출해 보안 쿠키(`session_token`)를 설정하도록 수정합니다.
- `getHostReservations(hostEmail)`: 세션 토큰의 이메일과 매개변수로 전달된 `hostEmail`이 일치하는지 검증합니다.
- `getSecurityReservations()` / `checkInVisitor()` / `checkOutVisitor()`: 세션 토큰의 이메일이 `SecurityStaff` 테이블에 존재하는 검증된 경비원인지 확인합니다.

#### [MODIFY] [lpr.ts](file:///e:/visit/src/app/actions/lpr.ts)
- `getEmployeeVehicles()`, `addEmployeeVehicle()`, `deleteEmployeeVehicle()`, `updateEmployeeVehicle()` 호출 시, 요청자가 등록된 보안 협력사 또는 관리자인지 서버 사이드 검증을 거친 후 DB 처리를 수행하도록 수정합니다.

#### [MODIFY] [smtp.ts](file:///e:/visit/src/app/actions/smtp.ts)
- 설정 변경 시 최고 관리자 토큰(`validateSecurityToken`) 검증 외에, 실제 설정 액션 시에도 보안 세션을 검증합니다.

## Verification Plan

### Automated Tests
- `npm run build`를 수행하여 TypeScript 컴파일 에러 및 린트 오류가 없는지 검증합니다.

### Manual Verification
1. **권한 우회 테스트**: 로그인하지 않은 상태에서 브라우저 F12 콘솔 또는 Postman 등을 통해 `getSecurityReservations` 및 `getEmployeeVehicles` 서버 액션을 임의 호출했을 때 데이터 유출이 차단되고 `Unauthorized` 에러가 발생하는지 확인합니다.
2. **쿠키 변조 테스트**: 임의로 `security_email` 쿠키를 다르게 생성하여 수동 변조하더라도 서버가 검증된 `session_token`을 기준으로만 응답하여 위변조가 통하지 않는지 확인합니다.
3. **정상 시나리오 테스트**: 이메일 링크를 통해 대시보드 로그인 완료 후 실시간 예약 확인 및 승인/반려 처리가 정상 작동하는지 흐름을 검증합니다.
